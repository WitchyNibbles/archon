# Archon Product Framing

Use when broad or ambiguous asks need product framing before architecture or implementation.

Goal: convert vague intent into a concrete product brief with clear scope and a smallest useful milestone.

1. Identify user, problem, value, and urgency.
2. Separate desired outcome from proposed solution.
3. Name explicit non-goals and scope boundaries.
4. Turn the ask into acceptance criteria and decision questions.
5. Pick the smallest useful milestone that can produce evidence.
6. Call out risky ambiguity early instead of hiding it inside planning.

## Rules

- do not confuse roadmap ideas with an approved milestone
- do not write slogans where acceptance criteria belong
- do not collapse product framing into architecture prematurely
- if external evidence matters, cite it and mark it current

## Output

Return user, problem, value, scope, non-goals, success criteria, and smallest useful milestone.
